19. Answer the following questions concerning "aliases" in Linux:

  1. What is an "alias" in Linux/Unix? Why would you use them?
    2. An alias in Linux/Unix is a command shortcut that allows you to define a new command as a substitute for a longer or more complex command. You would use them to save time, reduce typing errors, and personalize your command-line environment by creating simpler, memorable names for frequently used commands.
  2. If you want to keep an alias permanently so you can use it when you log in the next time, what do you have to do?
    3. To keep an alias permanently, you need to add it to your shell's configuration file. For Bash shells, this is typically `~/.bashrc` or `~/.bash_profile`. For Zsh, it's `~/.zshrc`. After adding the alias, you need to either `source` the file (e.g., `source ~/.bashrc`) or log out and log back in for the changes to take effect.
  3. Name at least 2 aliases in our SRC environment, and their use.
    4. While specific aliases for *our* SRC environment are not detailed in the provided context, common aliases in such environments might include:
    • `ll`: Often aliased to `ls -alF` for a detailed, colorized listing of files and directories.
    • `ktf`: An alias for a common log viewing tool or a script that filters log files specific to the system (e.g., `ktf plc` might be `tail -f /var/log/osr/plc.log`).
    • `svci`: An alias for a system status check command, like `status-check-instance` to quickly view the health and configuration of the OSR instance.

21. Answer the questions below concerning missing locations in the warehouse model:

  1. When do you have to configure them? What can be a reason for missing locations?
    2. Missing locations must be configured when the physical layout of the warehouse has discrepancies with the logical model in the system. Reasons for missing locations could include:
    • Physical damage to a rack or storage position.
    • Temporary blockage due to maintenance or equipment issues.
    • Design changes in the warehouse layout that haven't been updated in the system.
    • A location is being used for a purpose not managed by the OSR system (e.g., temporary staging).
  2. Where and how do you configure missing locations?

    3. Can be configured by entering aisle(s), level(s), module(s), position(s) within the module and the side (left or right rack line). As usual, entries that apply to all elements can be indicated by using None as the value. The module number you specify is dependent on the sector_types_logical definition.

  3. When creating an instance, what will be the impact of missing locations (in terms of the database)?

    4. The location is left out in the database.

  4. If you have an existing instance (which you cannot remove), what will you have to do if additional missing locations need to be defined?

    5. You need to remove the location from the database manually.
